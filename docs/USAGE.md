## 1. 安装

前往 [Releases](https://github.com/ZtestAi/z-switch/releases) 下载对应平台的安装包：

- **Windows**：`.exe`（NSIS 安装器）
- **macOS**：`.dmg`（区分 Apple Silicon 与 Intel，请按芯片选择）
- **Linux**：`.AppImage` 或 `.deb`

> 当前版本尚未做代码签名，Windows SmartScreen 可能提示「未知发布者」，macOS 首次打开需右键 →「打开」。这属于正常现象，详见 [FAQ](#14-常见问题-faq)。

## 2. 首次启动

首次运行时，z-switch 会自动完成三件事：

1. **保存原始配置** — 把本机现有的 Claude Code / Codex 配置完整快照到 `~/.z-switch/original/`，任何时候都能一键还原。
2. **建立官方账号卡片** — 为 Claude 和 Codex 各生成一张「官方账号」卡片，代表客户端本机登录状态。
3. **自动导入现有中转配置** — 若检测到你已经在用某个 API 中转站，会自动把它导入为一张供应商卡片并设为当前项。

## 3. 界面总览

- **顶部标签页**：在 **Claude Code** 与 **Codex** 两个分区之间切换，各自独立管理供应商。
- **右上角工具栏**：
  - **文档** — 打开本教程
  - **测速** — 一次性测试所有端点的 TCP 延迟
  - **导入 / 导出** — JSON 形式备份或迁移供应商
  - **设置** — 主题、开机自启、本地路由代理等
  - **添加** — 新建一张供应商卡片
- **供应商列表**：每张卡片显示名称、地址、模型/协议标签，以及切换、真实测试、复制、编辑、删除等操作；卡片可拖拽排序。

## 4. 添加供应商

点击右上角 **添加**，填写：

- **名称** — 便于识别的备注名
- **Base URL** — 中转站地址，支持智能推断（填入常见地址会自动补全）
- **API Key** — 你的密钥

### Claude 供应商

可分别配置：

- **主模型**
- **Haiku / Sonnet / Opus / Fable** 四个模型档位（分别对应不同用途，留空则用中转站默认）

### Codex 供应商

- 选择协议：**`responses`** 或 **`chat`**（按中转站支持情况选择）

保存前可先做 [连通性测试、模型拉取、测速](#6-供应商体检连通性--模型--测速--真实测试)。表单和 **JSON** 两种编辑方式可自由切换，适合高级用户直接粘贴配置。

## 5. 切换供应商

在目标卡片上点击 **切换**：

- **直连模式（默认）** 会立即把该供应商写入客户端配置文件。
- 切换**之前**，z-switch 会把当前生效（live）的配置**回填**到上一张卡片（backfill），避免你在配置文件里的手工修改丢失。

生效时机：

- **Claude Code** 通常在**下一次请求**时读取新配置，无需重启。
- **Codex CLI** 可能需要**重启**后生效。

## 6. 供应商体检：连通性 / 模型 / 测速 / 真实测试

每张卡片都提供以下能力：

- **连通性测试** — 检查地址与密钥是否可用。
- **模型列表拉取** — 从中转站拉取可用模型，便于选择档位。
- **TCP 测速** — 测量到端点的网络延迟。
- **真实测试** — 使用已保存的地址、密钥、模型和协议，向供应商发送一条最小的 `Hi` 请求（最多输出 32 tokens），在独立弹窗中实时显示**回复内容、首字耗时、总耗时**。

> ⚠️ 真实测试会产生**极少量**模型调用费用。测试内容、模型回复与密钥**不写入任何持久化文件**，结果仅在本次运行中回显到卡片。

## 7. 官方账号卡片

「官方账号」卡片代表**客户端本机登录**，不保存任何 API Key：

- **Claude** — 切到官方账号时，会清除中转相关的环境变量，回到客户端自身的登录状态。
- **Codex** — 切走前会保存客户端刷新后的登录态，切回时自动恢复。

因此你可以随时在「API 中转站」与「官方账号」之间来回切换，互不干扰。

## 8. 导入与导出

- **导出 JSON** — 把当前供应商列表备份成 JSON，用于迁移或存档。
- **导入 JSON** — 从 JSON 恢复供应商列表。

> 导入导出的是**供应商列表**，与首次启动保存的**原始配置快照**是两回事（后者用于「恢复原始配置」，见第 11 节）。

## 9. 本地路由代理（实验）

如果你希望在客户端**运行期间热切换**供应商、无需改配置或重启，可在 **设置** 中开启 **本地路由代理**。

开启后，z-switch 监听 `127.0.0.1:8899`（可在配置中覆盖），并把两个客户端的 Base URL 分别指向：

- `http://127.0.0.1:8899/claude`
- `http://127.0.0.1:8899/codex`

代理会根据当前选中的中转站注入鉴权信息并转发请求——**不修改请求体、不做协议转换、不记录用量**。切换中转站时会立即更新上游地址与鉴权。

**高级设置**（设置页）：连接 / 首段 / 流静默 / 非流式超时、请求体硬上限、连接池与 TCP Keepalive（流式返回与连接复用始终开启）。

**已知限制**：

- 代理**不会**同步改写已落入 live 配置的模型名和 Codex `wire_api`。因此热切换只适用于**模型 / 协议兼容**的供应商；跨模型或跨协议切换仍需**关闭代理后直连切换**。
- 目前**不提供**自动重试或故障转移。
- 该模式仍在持续验证中，建议对稳定性有要求的场景优先使用直连模式。

失败请求会写入 `~/.z-switch/logs/proxy-errors.jsonl`，仅记录上游状态、脱敏 URL、失败阶段和截断后的错误详情，**不记录请求正文**；日志会自动脱敏密钥并按文件大小轮转，可在设置页打开目录或清空。

## 10. 设置

- **主题** — 浅色 / 深色 / 跟随系统
- **开机自启**
- **本地路由代理** 及其高级参数（见第 9 节）
- **错误日志目录** — 一键打开或清空代理错误日志
- **恢复原始配置** — 分别还原 Claude Code 或 Codex（见第 11 节）

## 11. 恢复原始配置 / 解除管理

- **恢复原始配置** — 把首次启动时保存在 `~/.z-switch/original/` 的快照写回客户端。恢复**之前**仍会保存一份时间戳备份，双保险。
- **删除正在使用的供应商** 时可选择：
  - 恢复首次原始配置，或
  - 保留电脑当前配置、仅解除 z-switch 的管理关系。

## 12. 深链导入

z-switch 注册了 `zswitch://import` 深链。中转站或分享方可以生成这样的链接，用户点击后 z-switch 会自动拉起并导入对应配置，省去手工填写。

## 13. 文件位置

| 路径 | 内容 |
| --- | --- |
| `~/.claude/settings.json` | Claude Code 客户端配置（直连模式写入 `env`） |
| `~/.codex/auth.json`、`~/.codex/config.toml` | Codex 客户端配置（直连模式写入） |
| `~/.z-switch/original/` | 首次运行保存的原始配置快照 |
| `~/.z-switch/backups/` | 每次写入前的时间戳备份 |
| `~/.z-switch/logs/proxy-errors.jsonl` | 本地代理的脱敏错误日志 |

## 14. 常见问题 FAQ

**Q：Windows 提示「Windows 已保护你的电脑」/ macOS 说「无法打开」怎么办？**
A：当前版本未做代码签名。Windows 上点「更多信息 → 仍要运行」；macOS 上在应用图标右键 →「打开」，或到「系统设置 → 隐私与安全性」允许。后续版本会考虑加入代码签名。

**Q：切换后没生效？**
A：Claude Code 通常在下一次请求时生效；Codex CLI 需要重启。若仍无效，检查该卡片的 Base URL / 密钥是否正确，可用「连通性测试」验证。

**Q：我用第三方插件（如状态栏 ccline）改了 `~/.claude/settings.json`，切换供应商会覆盖它吗？**
A：**顶层字段（如 `statusLine`、`hooks`、`permissions` 等）始终保留**，z-switch 只替换 `env` 这一个顶层字段。但写在 `env` **内部**的第三方变量、以及 Codex `config.toml` 里的第三方段落（如 `[mcp_servers]`），目前是**按供应商卡片记忆**的：它们会跟随「切换离开时激活的那张卡片」，切到另一张卡片时可能暂时不在生效配置里（切回原卡片会恢复）。

**Q：真实测试会不会泄露我的密钥或对话？**
A：不会。测试内容、模型回复与 API Key 不写入任何持久化文件，只在本次运行内回显。

**Q：本地路由代理和直连模式该用哪个？**
A：日常推荐**直连模式**（稳定、零中间层）。只有当你需要在客户端运行期间频繁热切换、且供应商之间模型/协议兼容时，再考虑开启**本地路由代理**。

**Q：配置文件坏了会被 z-switch 清空吗？**
A：不会。如果现有配置文件不是合法 JSON，z-switch 会**中止写入并报错**，绝不覆盖，以保护你的原有配置。
